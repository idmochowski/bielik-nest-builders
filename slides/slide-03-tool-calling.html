Tool Calling Agent vs ReAct

Model używa natywnego API do wywoływania funkcji — strukturalny JSON zamiast parsowania tekstu.

  User: "Jaka jest pogoda w Warszawie?"
           │
           ▼
  ┌──────────────────────────────────────────────────┐
  │  LLM → JSON (natywne tool calling):              │
  │  {"name":"get_weather", "args":{"city":"Warszawa"}}│
  └──────────────────────┬───────────────────────────┘
                         │
           ┌─────────────▼─────────────────────┐
           │  Tool → {"temp": 12, "słonecznie"} │
           └─────────────┬─────────────────────┘
                         │
           ┌─────────────▼──────────────────────────┐
           │  LLM: W Warszawie jest 12°C, słonecznie.│
           └─────────────────────────────────────────┘

Zalety

  • Bardziej niezawodny — brak parsowania tekstu
  • Szybszy — brak narzutu promptu ReAct
  • Równoległe wywołania narzędzi

Wady

  • Wymaga wsparcia modelu + serwera (vLLM)
  • Mniej transparentny — rozumowanie ukryte
  • Mniejsze modele mogą mieć trudności
workflow:
  _type: tool_calling_agent
  tool_names: [wiki_search, current_datetime]
  llm_name: bielik

Wymaga vLLM skonfigurowanego z --enable-auto-tool-choice i customowym parserem narzędzi.
